## The bug
`createWallet`'s inline path persisted the active handle via `setActiveHandle`,
but four other success paths ran the ceremony and returned the result WITHOUT
persisting — leaving the SDK wallet store (`dexter:passkey:userHandle` +
roster) empty after a successful sign-in/create.

## Fix (TDD — red → green)

Re-located post-Task-1 line numbers before editing.

| # | Path | File / site | Persist |
|---|------|-------------|---------|
| 1 | `passkeyLogin` inline | `src/relay.ts` (`return submitLogin(...)`, was ~62) | `await`; `if (result.vault) setActiveHandle(vault.userHandle, undefined, vault.credentialId)` |
| 2 | `passkeyLogin` popup | `src/relay.ts` (`openCeremonyPopup<SignInResult>('signin')`, was ~40-45) | `await`; same guarded persist |
| 3 | `continueWithDexter` popup | `src/relay.ts` (`openCeremonyPopup<ContinueResult>('continue')`, was ~82-88) | `await`; `kind:'create'` → top-level `handle`/`config.name`/`credentialId`; `kind:'signin'` → guarded on `vault?.userHandle` |
| 4 | `createWallet` popup | `src/enroll.ts` (`openCeremonyPopup<CreateWalletResult>('create')`, was ~66-72 pre-Task-1) | `await`; persist `result.handle`/`config.name`/`result.credentialId` on the CALLER's origin |
